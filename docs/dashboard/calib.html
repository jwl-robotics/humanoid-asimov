▍ Stage 3 — Online Self-Calibration

While walking its loop, the robot estimates its own camera-mount rotation online — no calibration rig, only the onboard sensors the VIO already uses. It also carries an FD-verified camera–IMU time-offset state (td), but this constant-rate loop sits at that state's observability degeneracy, so td is held frozen here while the mount calibrates online. Validation: inject a known mount error and recover it.

The idea

A real robot's head camera is never mounted exactly where the CAD says, and its frame timestamps are a few ms out of sync with the IMU clock. The classic fix is a calibration rig (wave a checkerboard before deploying). In-situ self-calibration instead has the robot recover those errors itself, while just walking — robust to lifetime drift, no downtime.

The Stage-2 VIO filter is augmented with a camera-mount rotation extrinsic δφ_HC (3) and a time offset td (1); the error state grows 21 → 25, with the calibration states placed before the stochastic clone so cloning stays a clean truncate-then-append. Both enter the existing relative-rotation vision measurement, so no new sensor is needed.

The observability structure (the real lesson)

A pure yaw-loop observes only the mount component perpendicular to the turn axis. The component along the yaw axis is nearly invisible per frame (only the gait's ripple excites it), and the time offset is weakly observable too — the loop sits near the constant-angular-rate degeneracy where a common time shift cancels. This isn't a bug: the finite-difference check passes to ~1e-7, and the filter's covariance reports the anisotropy (the φ_y funnel stays ~6× wider than φ_x).

TakeawayA consistent filter recovers what the trajectory makes observable and honestly flags what it cannot — a stronger result than a lucky full recovery. Full 3-axis + time-offset calibration needs richer rotational excitation than a single yaw-loop.
